1990s — 2010s
Era 1 — Data Warehouse
What existed
Companies used Oracle, SQL Server, Teradata. Structured data only. One central database. IT team controlled everything. Very expensive — cost crores per year.
What was good
✓One place for all data
✓Very structured and organised
✓Access control existed
What was bad
✗Only structured data — no images, JSON, logs
✗Extremely expensive
✗Cannot scale for big data
Storage problem did NOT exist in warehouse era — because IT controlled everything strictly. Only one database, one team managing it.
2010 — 2016
Era 2 — Data Lake (THIS is when chaos started!)
What happened
AWS launched S3 in 2006. Companies realized — store EVERYTHING cheaply on S3! Images, JSON, logs, CSVs, videos — all dumped into S3. This was the Data Lake idea. No structure, no rules, just dump and store.
THIS is exactly when "data everywhere" chaos started!
✗Engineer 1 dumps files in s3://company/raw/2021/
✗Engineer 2 dumps files in s3://company/data/new/
✗Engineer 3 creates a new bucket s3://company-data-2/
✗No naming convention, no governance, no access control
✗6 months later — nobody knows where anything is!
Data Lake gave freedom — but too much freedom became chaos. S3 was cheap so everyone dumped data everywhere. This is why they called it "Data Swamp" not Data Lake!
2013 — 2021
Era 2.5 — Databricks launched but no storage standard
Databricks launched on top of this chaotic Data Lake
Databricks was built to process the messy Data Lake data using Spark. But engineers using Databricks had no clear standard for WHERE to save their processed output. So they used whatever was convenient.
What engineers did — and what went wrong
✗Some saved to cluster disk — data lost when cluster stopped
✗Some saved to DBFS — no security, anyone could see anything
✗Some saved to workspace files — hit 500MB limit on big data
✗Some saved directly to S3 — no tracking, no governance
✗Result — processed data was as chaotic as raw data!
Your question was perfect — YES this chaos happened DURING the Data Lake era, not before or after. Data Lake created freedom. Freedom created mess. Databricks inherited that mess.
2019
Era 3 — Delta Lake (partial solution)
What Delta Lake solved
Databricks created Delta Lake in 2019. It added ACID transactions on top of S3 — so data was reliable and not corrupted. But Delta Lake solved DATA QUALITY — not the storage chaos problem. Engineers still saved files wherever they wanted.
Delta Lake solved
✓Data corruption — ACID transactions
✓Data reliability — no partial writes
✓Time travel — rollback bad changes
Delta Lake did NOT solve
✗Where to store files — still scattered
✗Who can access what — no governance
✗Audit trail — still missing
Delta Lake = data is now reliable. But storage is still chaotic. You have clean data — but nobody knows where it is or who can see it!
2021
Era 4 — Lakehouse (architecture solution)
What Lakehouse solved
Databricks coined "Lakehouse" — combine the reliability of Data Warehouse with the flexibility of Data Lake. Delta Lake + Spark + one platform = Lakehouse. But storage governance was STILL not fully solved. Engineers still had multiple storage options and no clear rule.
The remaining problem
Lakehouse told you HOW to process data — Bronze, Silver, Gold layers. But it still did not tell every engineer WHERE exactly to save files and HOW to control who sees what. The slide you saw lists all the options that existed — and the confusion they caused.
Lakehouse = great architecture. But the STORAGE GOVERNANCE problem was still not 100% solved. That is what Unity Catalog came to fix.
2022 — Today
Era 5 — Unity Catalog (final solution!)
What Unity Catalog finally solved
Unity Catalog launched in 2022. It gave ONE answer to every storage question — use Volumes. It brought governance, security, audit trail, and access control in one place. Now every engineer in every company has one clear rule: store in Unity Catalog Volumes.
Complete problem → solution journey
✓Data Lake chaos (2010) → Lakehouse architecture (2021)
✓Data corruption (2013) → Delta Lake ACID (2019)
✓Storage chaos (2013-2021) → Unity Catalog Volumes (2022)
✓No governance → Unity Catalog fine-grained access control
✓No audit trail → Unity Catalog logs every read and write
So your original understanding was correct — in a properly set up Lakehouse TODAY, data lives in S3/ADLS accessed through Unity Catalog Volumes. The chaos happened DURING the transition from Data Lake to Lakehouse between 2013 and 2022.
