Step 1 — Admin creates pool
Admin → Databricks UI
Admin opens Databricks → Compute → Pools → clicks Create Pool. Sets: min 5 machines, machine type, cloud region.
↓
Databricks UI → Control Plane
UI sends pool config to Databricks Control Plane (the brain of Databricks). Control Plane saves pool settings in its internal database.
↓
Control Plane → Cloud Provider (AWS / Azure / GCP)
Control Plane immediately calls Cloud API — "start 5 VMs of this type in this region." Cloud starts the VMs and keeps them running.
↓
VMs → Pool registry
Once VMs are running, Control Plane registers them as "idle" in the pool. They sit and wait — doing nothing, costing money, but saving time!
At this point — 5 machines are ON, idle, registered in pool. No cluster yet.
Step 2 — Engineer creates cluster
Engineer → Databricks UI
Engineer goes to Compute → Create Cluster → selects "use pool" → picks the pool Admin created → saves cluster config.
↓
Databricks UI → Control Plane
Control Plane saves cluster config with pool reference attached. Cluster is NOT started yet — just configured. Pool is just linked, not consumed.
At this point — cluster config saved. Pool still has all 5 idle machines. Nothing consumed yet.
Step 3 — Cluster starts
Engineer clicks Start / Job triggers cluster
Engineer clicks "Start cluster" or a scheduled job triggers it automatically.
↓
Control Plane checks pool
Control Plane looks at pool registry — "how many idle machines available?" Finds M1, M2, M3, M4, M5 all idle. Cluster needs 3 → takes M1, M2, M3.
↓
Control Plane → M1, M2, M3
Sends Spark install command to M1, M2, M3. Since VMs are already running — Spark installs in seconds. Driver starts on M1. M2, M3 become workers.
↓
Driver (M1) → Workers (M2, M3)
Driver registers workers. Cluster is fully READY. M4 and M5 stay idle in pool — available for another cluster.
Total time = 30 seconds. Without pool this same step takes 5 to 7 minutes!
Step 4 — Job runs
Engineer → Notebook / Job
Engineer runs a notebook or scheduled job. Code like df.read.format("orc") or SELECT AVG(salary) is submitted to the cluster.
↓
Driver (M1) → Spark engine
Driver receives the query. Spark engine on Driver creates an execution plan — decides how to split work across workers.
↓
Driver → Workers (M2, M3)
Driver sends tasks to M2 and M3. Each worker reads its assigned data partition from Databricks storage (DBFS / Delta / ORC files).
↓
Workers → Driver — results back
Workers process data and send results back to Driver. Driver combines results and returns final output to Engineer's notebook.
During this whole time — M4 and M5 are still sitting idle in pool, ready for another engineer's cluster!
Step 5 — Cluster terminates
Job finishes or timeout hits
Job completes OR auto-termination kicks in (e.g. idle for 30 mins). Cluster shutdown is triggered.
↓
Control Plane → M1, M2, M3
Control Plane sends "stop Spark" command to M1, M2, M3. Spark processes shut down. But the VMs themselves are NOT terminated — just cleaned up.
↓
M1, M2, M3 → Pool registry
Control Plane marks M1, M2, M3 as "idle" again in the pool registry. Now pool has all 5 machines idle again — M1 M2 M3 M4 M5.
↓
Pool — fully restored
Next cluster that starts will again get machines from pool instantly. The cycle repeats — no cloud provisioning needed!
VMs are never deleted — only returned. This is the whole magic of pools. Cloud cost continues for idle VMs but cluster start time = nearly zero.
