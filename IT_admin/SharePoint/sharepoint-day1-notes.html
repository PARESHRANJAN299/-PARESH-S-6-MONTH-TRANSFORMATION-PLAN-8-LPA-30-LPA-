1. SharePoint Online vs On-Premises Architecture

SharePoint Online (Cloud-based)

Key Characteristics:
  • Hosted by Microsoft - No server management required
  • Automatic updates - Always on the latest version
  • Subscription-based - Part of Microsoft 365
  • Elastic storage - Storage scales with licenses
  • Global availability - Accessible from anywhere with internet
  • Built-in security - Microsoft handles infrastructure security

Architecture Components

  • Multi-tenant architecture (shared infrastructure, isolated data)
  • Integrated with Microsoft 365 services (Teams, OneDrive, Outlook)
  • Modern experience focus (responsive, mobile-friendly)
  • Cloud-based apps and workflows (Power Automate, Power Apps)

Advantages

  • No infrastructure costs or maintenance
  • Rapid deployment and scalability
  • Regular feature updates
  • High availability and disaster recovery built-in
  • Reduced IT overhead

Limitations

  • Internet dependency
  • Limited customization compared to on-premises
  • Compliance requirements may restrict cloud adoption
  • Less control over infrastructure

SharePoint On-Premises (Server-based)

Key Characteristics:
  • Self-hosted - Runs on your own servers
  • Full control - Complete control over hardware, software, updates
  • One-time licensing - Purchase SharePoint Server license + CALs
  • Manual updates - IT team controls update schedule
  • On-site or data center - Data stays within your infrastructure

Architecture Components

  • Farm architecture (Web Front End, Application, Database servers)
  • Classic experience primary (modern experience limited)
  • Custom solutions and farm solutions supported
  • SharePoint Designer workflows

Advantages

  • Complete control over environment
  • Full customization capabilities
  • Data sovereignty (stays on your servers)
  • Can work without internet connectivity
  • Legacy system integration easier

Limitations

  • High infrastructure and maintenance costs
  • Requires dedicated IT staff
  • Manual updates and patches
  • Scalability requires hardware investment
  • Disaster recovery is your responsibility

SharePoint Hybrid (Best of Both Worlds)

What is Hybrid?
Combination of SharePoint Online and On-Premises that allows gradual migration to cloud while maintaining on-premises investment.

Common Hybrid Scenarios

  • Hybrid Search - Unified search across both environments
  • Hybrid OneDrive - Users get OneDrive in cloud while using on-premises SharePoint
  • Hybrid Sites - Redirects My Sites to SharePoint Online
  • Hybrid Taxonomy - Shared term store across environments

2. Tenant, Site Collection, and Site Hierarchy

Tenant Level (Top of Hierarchy)

What is a Tenant?
Your organization's entire SharePoint Online instance (e.g., contoso.sharepoint.com). Single tenant per Microsoft 365 subscription containing all site collections for your organization.

Tenant-level Settings

  • Global sharing policies
  • Default storage limits
  • External sharing controls
  • Organization-wide branding
  • Tenant app catalog
  • Data loss prevention (DLP) policies

Managed by: Global administrators and SharePoint administrators

Site Collection Level (Second Layer)

What is a Site Collection?
Independent container of sites with shared characteristics. Security boundary where permissions don't cross site collections.

Site Collection Characteristics

  • Has own URL: https://contoso.sharepoint.com/sites/marketing
  • Storage quota allocated from tenant pool
  • Own recycle bin, audit logs, and settings
  • Isolation boundary - Content and permissions are separate
  • Own governance - Unique retention, compliance policies
  • Recovery unit - Can restore entire site collection

Site Collection Contains

  • Root site (top-level site)
  • Subsites (optional child sites)
  • Document libraries and lists
  • Site collection features
  • Site collection administrators
When to Create New Site Collection:
  • Different departments with separate security needs
  • Projects requiring isolated permissions
  • Different compliance or retention requirements
  • Need for separate branding or customization
  • Distinct business units or teams

Site Level (Third Layer)

Team Sites

  • Collaboration-focused
  • Connected to Microsoft 365 Group
  • Includes Teams integration, shared mailbox, calendar
  • Default document library
  • Members can contribute content

Communication Sites

  • Broadcasting information (intranet pages, news, announcements)
  • Not connected to Microsoft 365 Group
  • Designed for few authors, many readers
  • Rich publishing capabilities
  • Modern web parts and layouts
Important Note: Microsoft recommends flat architecture (no subsites) in modern SharePoint. Use hub sites for organization instead.

3. Hub Sites and Site Organization

What are Hub Sites?

Definition:
Hub sites connect related sites together for unified navigation, branding, and search. They are NOT a security boundary, just organizational grouping where sites associate with hubs voluntarily.

Key Benefits

  • Unified navigation - Shared top navigation across associated sites
  • Common branding - Consistent look and feel
  • Aggregated content - Roll-up news, events, activities
  • Unified search - Search across all hub-associated sites
  • Flexibility - Sites can join/leave hubs easily

Hub Site Architecture

Hub Site (Marketing Hub) │ ├── Associated Site 1 (Product Marketing) ├── Associated Site 2 (Content Marketing) ├── Associated Site 3 (Events) └── Associated Site 4 (Brand Guidelines)
Important Notes:
  • Hub sites are regular site collections registered as hubs
  • Sites maintain their own permissions
  • Can have up to 2,000 sites associated with one hub
  • Hub site navigation appears on all associated sites

When to Use Hub Sites

  • Organizing department sites (HR hub with Recruitment, Benefits, Training sites)
  • Project portfolios (Project hub with individual project sites)
  • Regional organization (EMEA hub with country-specific sites)
  • Topic-based organization (Product hub with different product sites)

Planning Hub Site Structure

  1. Identify logical groupings - What sites naturally belong together?
  2. Define hub owners - Who manages the hub navigation and branding?
  3. Plan navigation - What links should appear in hub navigation?
  4. Consider search scope - Do users need to search across these sites together?
  5. Branding consistency - Should these sites share theme and logo?

Modern vs Classic Site Organization

Classic Approach (Deprecated) Modern Approach (Recommended)
Heavy use of subsites Flat architecture - all sites at root level
Deep hierarchies (site → subsite → sub-subsite) Hub sites for organization
Complicated permission inheritance Clear permissions - each site manages its own
Difficult to reorganize Flexible - easy to reorganize by changing hub associations

4. Storage and Governance

Storage Architecture

Tenant Storage Pool

Formula: Base storage = 1 TB + 10 GB per licensed user
Example: 100 users = 1 TB + 1 GB (100 × 10 GB) = 2 TB total
  • Shared pool across all site collections
  • Additional storage can be purchased

Site Collection Storage

  • Default: Each new site gets storage from tenant pool
  • Can set custom storage quotas per site collection
  • Storage warnings at 90% capacity
  • Can reallocate storage between site collections

Storage Management

  • Active Storage admin center - Monitor usage across tenant
  • Site storage metrics - See what's consuming space
  • Recycle bins - Two-stage (site and site collection level)
  • Version history - Consumes storage (can limit versions)
Storage Best Practices:
  • Monitor storage usage regularly
  • Set appropriate quotas based on business needs
  • Enable versioning limits (e.g., keep last 50 versions)
  • Clean up unused sites periodically
  • Use retention policies to auto-delete old content

Governance Framework

What is Governance?
Policies, procedures, and controls for managing SharePoint that ensure security, compliance, and efficient usage while balancing user freedom with organizational control.

1. Site Provisioning Governance

Who can create sites?

  • Option 1: Anyone (self-service) - Most flexible, requires monitoring
  • Option 2: IT only - Most controlled, can create bottleneck
  • Option 3: Designated owners - Balanced approach

Site Lifecycle:

  • Creation - Who, when, how
  • Active use - Regular review of activity
  • Archival - Inactive site handling (read-only)
  • Deletion - Cleanup of unused sites (93-day recycle bin)

2. Security and Compliance Governance

Access Control:

  • External sharing policies (disabled, existing guests, new and existing, anyone)
  • Conditional access (location, device-based access)
  • Multi-factor authentication requirements
  • Privileged access management

Data Protection:

  • Sensitivity labels - Classify and protect content (Confidential, Internal, Public)
  • Data Loss Prevention (DLP) - Prevent sensitive data leaks
  • Retention policies - Keep data for required period, delete after
  • eDiscovery - Legal hold and search capabilities

3. Information Architecture Governance

Content Organization:

  • Hub site structure
  • Site and library naming conventions
  • Metadata standards (required columns)
  • Content types across organization

4. User Experience Governance

Branding and Customization:

  • Organization-wide themes
  • Hub site themes
  • Approved web parts and extensions
  • Custom development guidelines

5. Permission Models

SharePoint Permission Levels (Default)

Permission Level Can Do Typical Use
Full Control Everything including changing permissions Site collection admins
Design Create lists/libraries, edit pages, apply themes Site designers
Edit Add, edit, delete items and documents Power users
Contribute Add, edit, delete own items, view all items Team members
Read View pages and items, download documents General viewers
View Only View items but cannot download Restricted viewing
Limited Access Access specific items (system-managed) Broken inheritance scenarios

SharePoint Groups

Default Groups (Created with each site):
  1. Owners - Full Control permission level
  2. Members - Edit or Contribute permission level
  3. Visitors - Read permission level
Best Practice: Always assign permissions to groups, not individuals

Permission Inheritance

Site Collection ↓ (inherits) Site ↓ (inherits) Document Library ↓ (inherits) Folder ↓ (inherits) Document

Inheritance Rules

  • Child objects inherit permissions from parent by default
  • Simplifies permission management
  • Reduces administrative overhead
  • Easier to audit

Breaking Inheritance

When to Break:

  • Confidential content requiring restricted access
  • Department-specific folders in shared library
  • Sensitive documents
  • Project-specific security needs
Consequences of Breaking:
  • Creates unique permissions (no longer inherits)
  • More complex to manage
  • Harder to audit
  • Can lead to permission sprawl

Permission Best Practices

  1. Use Groups, Not Individuals - Easier to manage, clearer security model
  2. Follow Least Privilege Principle - Give minimum permissions needed
  3. Minimize Permission Breaks - Keep inheritance where possible
  4. Regular Audits - Review who has access quarterly
  5. Document Your Model - Standard operating procedures
  6. External Sharing Controls - Set appropriate tenant-level policies
  7. Use Sensitivity Labels - Automatic permission application

6. Site Collection Creation and Management

Creating Site Collections

Method 1: SharePoint Admin Center (GUI)

Steps:
  1. Go to SharePoint Admin Center (admin.microsoft.com → SharePoint)
  2. Click Sites → Active sites
  3. Click + Create
  4. Choose site type: Team site or Communication site
  5. Fill in details: Site name, administrator, language, time zone
  6. Set storage quota (optional)
  7. Review and create

Method 2: PowerShell (Automation)

Benefits: Bulk creation, scripted provisioning, custom configurations, automation

# Connect to SharePoint Online
Connect-SPOService -Url https://contoso-admin.sharepoint.com

# Create Communication Site
New-SPOSite -Url https://contoso.sharepoint.com/sites/marketing `
            -Owner admin@contoso.com `
            -StorageQuota 1024 `
            -Title "Marketing Site" `
            -Template SITEPAGEPUBLISHING#0

# Create Team Site (no group)
New-SPOSite -Url https://contoso.sharepoint.com/sites/project1 `
            -Owner admin@contoso.com `
            -StorageQuota 2048 `
            -Title "Project Alpha" `
            -Template STS#3

Method 3: Self-Service (End Users)

Governance Considerations:
  • Can lead to site sprawl
  • Requires monitoring
  • Benefits: Faster provisioning, user empowerment
  • Risks: Unused sites, inconsistent naming

Site Collection Configuration

Storage Management

  • Set storage limits (manual or automatic)
  • Set warning threshold (e.g., 90%)
  • Monitor current usage and growth trends

Site Collection Administrators

Admin Capabilities:

  • Full control of site collection
  • Access all content (even if not explicitly shared)
  • Manage site collection features
  • Restore from recycle bin (both stages)
  • Access site collection audit logs

Site Collection Policies

Sharing Policies

Level Options
Tenant Level Anyone, New and existing guests, Existing guests only, Only people in organization
Site Level Can inherit from tenant OR be more restrictive (cannot be more permissive)

Access Control

  • Unmanaged devices (block, limited access, full access)
